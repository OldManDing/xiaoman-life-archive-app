## 当前阻断问题

### P0：CI/本地回归基线不能原样通过

1. `regression:baseline` 依赖 Prisma Client，但 CI 只执行 `npm ci`，没有执行 `npm run prisma:generate`。
2. API e2e 测试依赖 `JWT_ACCESS_SECRET` / `JWT_REFRESH_SECRET` 等测试环境变量，但这些变量在 spec 的 `beforeAll` 中设置过晚，`AppModule` 顶层 import 时配置校验已经执行。

修复建议：

- 在 CI 的 regression 前增加 `npm run prisma:generate`，或把它放进根级 `regression:baseline` 前置步骤。
- 在 `apps/api/jest.config.js` 增加 `setupFiles`，用独立 setup 文件在任何测试模块 import 前设置测试环境变量。
- API e2e 的 `afterAll` 增加 `if (app) await app.close()`，避免初始化失败时再抛二次错误掩盖根因。
- 用 Node 20 复跑一次最终门禁，保持和 CI 一致；可以补 `.nvmrc` 或 README 运行说明。

### P0：生产依赖存在 high 漏洞

生产依赖 audit 发现 high 风险集中在：

- `react-router-dom` / `@remix-run/router`
- `axios`
- `fast-xml-builder` / `fast-xml-parser`
- `lodash`（经 `@nestjs/config`）
- `multer`（经 `@nestjs/platform-express`）
- `tar`（经 `@mapbox/node-pre-gyp`）

修复建议：

- 先执行非破坏性升级：`npm audit fix`，重点确认 `axios`、`react-router-dom`、AWS XML builder、`bullmq/uuid`、`tar` 是否能在当前主版本内修复。
- 对需要 breaking change 的 Nest 相关项单独开升级分支，优先尝试 Nest 10 可用补丁版本；如果 audit 只能通过 Nest 11 修复，需要做兼容性评估后再升级。
- 升级后必须跑：`npm run prisma:generate`、完整回归基线、三端 build、Admin lint。

### P1：Admin lint 失败

失败点：

- `apps/admin/src/pages/list-pages.tsx` 直接修改 `state.result.list`，违反 React Hooks immutability 规则。
- `apps/admin/src/pages/index.tsx` 使用 `export *`，Fast Refresh 规则无法验证只导出组件。
- `apps/admin/src/pages/shared.tsx` 同时导出 hook/工具函数和组件，违反 Fast Refresh 推荐结构。

修复建议：

- 给 `useAdminListPage` 增加受控更新能力，例如返回 `setResult` 或 `updateResult`，页面通过 immutable updater 更新列表。
- 拆分 `pages/shared.tsx`：hook 和纯函数放到 `.ts` 文件，组件保留在 `.tsx` 文件。
- `pages/index.tsx` 改为显式导出，或拆分 barrel，避免 `export *` 触发 Fast Refresh 规则。
- 将 `npm run lint -w apps/admin` 接入根级回归门禁，修完后防止回归。

## 开发计划

### 第 1 阶段：修复质量门禁

目标：让 CI 和本地一条命令可稳定复现通过。

1. 修复 Prisma Client 生成步骤。
2. 修复 API Jest 测试环境变量加载时机。
3. 修复 API e2e 初始化失败后的二次 `app.close()` 错误。
4. 修复 Admin lint 5 个错误。
5. 把 Admin lint 纳入根级 `regression:baseline` 或新增 `quality:check`。

验收命令：

```powershell
npm run prisma:generate
npm run regression:baseline
npm run build:api
npm run build:web
npm run build:admin
npm run lint -w apps/admin
```

### 第 2 阶段：依赖安全修复

目标：消除生产依赖 high 漏洞，避免一次性强升造成业务回归。

1. 执行非破坏性 `npm audit fix` 并复跑门禁。
2. 对剩余 Nest/Multer/Lodash 链路做单独升级验证。
3. 记录无法立即修复的漏洞、影响面和临时缓解措施。

验收命令：

```powershell
npm audit --omit=dev --audit-level=moderate
npm run regression:baseline
npm run build:api
npm run build:web
npm run build:admin
```

### 第 3 阶段：补齐测试覆盖

目标：现有测试能验证基础壳层和 API smoke，但前端页面、权限矩阵、异常路径覆盖偏薄。

1. Web：补登录态、家庭、时间线、记录创建/详情、个人页关键交互测试。
2. Admin：补登录、用户禁用/启用、媒体上传 token、审计日志列表、权限拒绝状态测试。
3. API：把 smoke 中的关键业务断言拆出更多单元/集成测试，覆盖鉴权失败、角色拒绝、无效输入、重复提交、分页边界。
4. 增加 coverage 脚本和最低覆盖率基线，先以当前项目状态建立可达标基线，再逐步提高。

### 第 4 阶段：产品开发优先级

建议先在质量门禁稳定后推进业务开发，优先顺序如下：

1. 用户端核心闭环：注册/登录、家庭成员、孩子档案、记录创建、时间线浏览。
2. 管理端最小运营闭环：用户列表、禁用/启用、内容/媒体审核、审计日志。
3. 媒体与 AI 异步任务闭环：上传确认、预览 URL、AI job 状态轮询、失败重试。
4. 发布准备：环境变量模板拆分、部署手册校准、数据库迁移/种子数据流程、生产可观测性。
