# 实施版：生产部署运行手册

更新时间：2026-05-11

## 1. 发布前门禁

必须在发布分支执行：

```bash
npm ci
npm run verify:production-template
npm audit --omit=dev --audit-level=moderate
npm run regression:baseline
npm run build:api
npm run build:web
npm run build:admin
npm run test:e2e
```

GitHub Actions 已配置同等门禁，并包含 MySQL 服务、生产模板校验、依赖审计、构建和 App/Admin E2E。

## 2. 服务器与目录

推荐单机 Docker Compose 首发：

- 代码目录：`/srv/xiaoman/xiaoman-life-archive-app`
- 环境变量：`/srv/xiaoman/xiaoman-life-archive-app/.env.production`
- TLS 证书：`/srv/xiaoman/xiaoman-life-archive-app/deploy/certs/fullchain.pem`
- TLS 私钥：`/srv/xiaoman/xiaoman-life-archive-app/deploy/certs/privkey.pem`
- MySQL 备份：`/srv/xiaoman/xiaoman-life-archive-app/deploy/backups/mysql`
- 容器日志：`docker compose logs`

当前线上实操路径为 `/opt/xiaoman-life/app/releases/<release>`，运行中的 API compose project 为 `xiaomanlife`，环境文件为当前 release 下的 `.env.server`；更新 API-only release 配置时需显式带 `-p xiaomanlife`，避免误创建新的 MySQL/Redis project。

## 3. 域名与反代

生产建议使用三个域名：

- App：原生 App 商店分发；隐私政策、帮助与后台入口复用 `https://nianlun.xmlga.top`
- 管理后台：`https://nianlun.xmlga.top`
- API：`https://webapi.xmlga.top/api/v1`

上线时确认 `deploy/nginx/reverse-proxy.conf` 与 `.env.production` 中域名一致：

```env
APP_BASE_URL=https://webapi.xmlga.top
VITE_API_BASE_URL=https://webapi.xmlga.top/api/v1
CORS_ORIGINS=https://nianlun.xmlga.top
```

## 4. 环境变量闭环

1. 复制 `.env.example` 为 `.env.production`。
2. 替换所有 `<...>` 占位符。
3. 生产禁止以下配置：
   - `JWT_ACCESS_SECRET=replace_me_access_secret`
   - `JWT_REFRESH_SECRET=replace_me_refresh_secret`
   - `JWT_ACCESS_SECRET` 与 `JWT_REFRESH_SECRET` 相同或任一长度少于 32 个字符
   - `ADMIN_INITIAL_PASSWORD=ChangeMe123!`
   - `STORAGE_PROVIDER=mock`
   - `AI_PROVIDER=mock`
   - `AI_BASE_URL` 指向 coding API，或未指向兼容 `/chat/completions` 的 API 根地址
   - `localhost` 或 `127.0.0.1` 出现在生产 URL 中
4. 执行：

```bash
npm run verify:production-env
```

`verify:production-env` 在真实 env 模式下会默认使用对象存储签名上传 URL 写入一段临时对象，检查浏览器上传所需 CORS 预检，再通过签名读取 URL 读回并清理临时对象；同时请求 AI `/chat/completions` 和高德 POI 接口，验证 `STORAGE_*`、`AI_API_KEY`、`AI_MODEL`、`MAP_API_KEY` 是否真实可用。如果对象存储签名上传、CORS、签名读取或清理失败，AI 返回 `UnsupportedModel` / `InvalidEndpointOrModel.NotFound`，或高德返回 `INVALID_USER_KEY`、超时、没有真实 POI 结果，不能继续发布；只有离线排查配置格式时才允许临时设置 `VERIFY_PRODUCTION_EXTERNAL_CHECKS=0`，正式上线验收不得跳过外部 provider 检查。

如果线上 release 已经部署，但 `verify:live-readiness` 失败在 AI 或地图 POI provider，拿到真实可用的 chat-completions 兼容 AI endpoint/model/key 或高德 Web 服务 Key 后，按最小变更原则人工更新当前 release 的 `.env.server`。不要在服务器落地执行临时 provider 配置脚本。

```bash
cp .env.server ".env.server.backup-$(date +%Y%m%d%H%M%S)"
chmod 600 .env.server.backup-*
# 按本次修复范围只更新必要变量：
# AI_PROVIDER=openai-compatible
# AI_BASE_URL=<chat_completions_api_root>
# AI_MODEL=<real_chat_model_or_endpoint_id>
# AI_API_KEY=<real_ai_key>
# MAP_PROVIDER=amap
# MAP_API_KEY=<real_amap_web_service_key>
# MAP_AMAP_ENDPOINT=https://restapi.amap.com/v3/place/text
# MAP_REQUEST_TIMEOUT_MS=5000
docker compose -p xiaomanlife --env-file .env.server -f docker-compose.api.yml config --quiet
docker compose -p xiaomanlife --env-file .env.server -f docker-compose.api.yml up -d --no-build api
```

更新时不得打印真实密钥；只在服务器环境文件中保存密钥，并保留 `.env.server.backup-<timestamp>` 备份。执行后仍必须从本地或 CI 重新跑：

```bash
LIVE_TEST_USER_CREDENTIAL=<smoke-user> LIVE_TEST_USER_PASSWORD=<smoke-password> npm run verify:live-readiness
```

若高德 Web 服务 Key 尚未申请完成，可把 `P1-03 地点真实 POI` 作为当前版本延期项处理，并显式运行：

```bash
LIVE_READINESS_ALLOW_P1_DEFERRALS=1 LIVE_TEST_USER_CREDENTIAL=<smoke-user> LIVE_TEST_USER_PASSWORD=<smoke-password> npm run verify:live-readiness
```

该模式只允许 P1/P2 延期项得到“条件通过”报告，P0 失败仍会退出失败；后台仍会展示“延期验收项”和后续 owner。高德 Key 申请完成后，必须去掉 `LIVE_READINESS_ALLOW_P1_DEFERRALS` 重新执行完整 live readiness，直到 `poi` 返回 `source=amap` 的文本 POI 候选。

## 5. 真实 Provider 验证

上线前必须在预发或生产灰度环境完成：

| 能力 | 验证方式 | 通过标准 |
| --- | --- | --- |
| 对象存储 | App 上传一张 JPG/PNG/WebP | 返回真实签名上传地址，上传成功，详情页可通过签名地址预览 |
| AI | 登录后调用 AI 预览，或创建记录后触发标题、摘要或标签任务 | 预览返回真实标题、摘要或标签；任务状态从处理中变为成功，输出字段不是 mock 固定文案 |
| 地图 POI | 在发布页输入地点关键字，如“儿童公园” | 返回真实 POI 候选，选择后保存到记录详情和时间轴 |
| 后台审计 | 后台执行冻结、下架、媒体审核、AI 重试/取消 | 审计日志记录操作者、动作、目标对象、原因和请求来源 |

短信能力当前默认关闭。若后续重新启用短信，应设置 `SMS_ENABLED=true`，并在预发环境验证真实短信签名、模板、频控和日志。
保持 `SMS_ENABLED=false` 时，`SMS_SIGN_NAME`、`SMS_TEMPLATE_CODE` 和短信密钥可以留空，生产健康检查会显示短信能力为 `disabled`。

## 6. 部署命令

首次部署：

```bash
cp .env.example .env.production
npm run verify:production-env
docker compose --env-file .env.production -f deploy/docker-compose.prod.yml build
docker compose --env-file .env.production -f deploy/docker-compose.prod.yml --profile migrate run --rm migrate
docker compose --env-file .env.production -f deploy/docker-compose.prod.yml up -d
```

后续发布：

```bash
git fetch --all --prune
git checkout <release-tag-or-commit>
npm run verify:production-env
docker compose --env-file .env.production -f deploy/docker-compose.prod.yml build
docker compose --env-file .env.production -f deploy/docker-compose.prod.yml --profile migrate run --rm migrate
docker compose --env-file .env.production -f deploy/docker-compose.prod.yml up -d
```

## 7. 健康检查

发布后验证：

```bash
curl -fsS https://webapi.xmlga.top/api/v1/health
curl -I https://nianlun.xmlga.top
docker compose --env-file .env.production -f deploy/docker-compose.prod.yml ps
API_BASE_URL=https://webapi.xmlga.top APP_BASE_URL=https://nianlun.xmlga.top ADMIN_BASE_URL=https://nianlun.xmlga.top deploy/ops/health-check.sh
LIVE_TEST_USER_CREDENTIAL=<smoke-user> LIVE_TEST_USER_PASSWORD=<smoke-password> npm run verify:live-readiness
```

API 健康检查会返回数据库状态、运行环境和当前 provider 名称，不返回密钥。
其中 `providers.map` 应为 `amap`；如果仍为 `mock` 或 `disabled`，不能宣称真实地点搜索已完成。
`verify:live-readiness` 会同时检查 API/App/Admin 可达性、安全响应头、CORS、数据库状态、`storage` / `ai` / `map` provider，以及登录后的真实 AMap POI 搜索结果，避免只凭 HTTP 200 或 provider 名称误判上线就绪。

## 8. 日志与审计

- API 容器日志：`docker compose --env-file .env.production -f deploy/docker-compose.prod.yml logs -f api`
- Nginx 日志：`docker compose --env-file .env.production -f deploy/docker-compose.prod.yml logs -f nginx`
- 后台审计日志：管理后台“审计日志”页面
- 错误排查优先级：API 5xx、短信失败、对象存储签名失败、AI 任务失败、后台写操作失败

## 9. 数据库备份

每日备份命令示例：

```bash
deploy/ops/mysql-backup.sh
```

建议至少保留 7 天日备份、4 周周备份；V1 发布验收口径要求后台“系统配置”页中的备份保留周期、恢复手册地址和最近恢复演练时间与实际运维策略一致。恢复前必须先停止 API 写入，恢复后执行健康检查和核心路径 E2E。恢复命令：

```bash
deploy/ops/mysql-restore.sh deploy/backups/mysql/<backup-file>.sql
```

## 10. 回滚方式

1. 记录当前 commit、镜像 tag 和数据库备份文件。
2. 若只回滚代码：

```bash
git checkout <previous-release-tag-or-commit>
docker compose --env-file .env.production -f deploy/docker-compose.prod.yml build
docker compose --env-file .env.production -f deploy/docker-compose.prod.yml up -d
```

3. 若涉及数据库结构回滚，先恢复发布前备份，再启动上一版本镜像。
4. 回滚后必须重新执行健康检查、登录、上传、AI 任务、后台审计抽检。

## 11. 监控告警

首发至少配置：

- API 健康检查失败告警
- API 5xx 错误率告警
- MySQL 容器异常退出告警
- 磁盘剩余空间告警
- 短信发送失败告警
- AI 任务失败率告警
- 对象存储上传失败告警

监控平台可使用云厂商监控、Uptime Kuma、Prometheus/Grafana 或现有运维平台，告警联系人和升级路径应在上线前确认，并在后台“系统配置”页维护当前联系人与联系方式；每次调整都会进入审计日志。
