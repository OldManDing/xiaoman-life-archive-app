# 实施计划：设计对齐与验收闭环

## 目标

将当前“可联调骨架”推进到更接近需求文档、设计图和 Figma 代码的 MVP 验收形态。

本轮重点不是重写架构，而是在保留现有 React + NestJS + Prisma 主体的前提下，修复关键链路问题、收敛占位内容、统一用户端移动 UI 语言，并补齐可执行验证。

## 当前偏差归纳

1. 用户端底部导航、页面外壳和通用标题卡片偏工程化，未贴近 Figma 的移动 App 视觉。
2. 首页快捷入口数量与设计不一致，首屏节奏有偏差。
3. 登录页缺少协议勾选，且默认保留演示手机号和验证码。
4. 上传媒体后未随记录提交，导致记录详情无法展示已上传媒体。
5. 禁用用户和禁用管理员仍可能继续登录或访问。
6. 家庭动态、草稿箱、月报纪念册、协议页等存在硬编码或占位伪完整。
7. README 与环境示例存在配置/文档引用漂移。

结论：

- API e2e/smoke 已闭环，不再作为未完成风险。
- 本地运行 API 前需确保 Docker Desktop 已启动，并执行 `docker compose up -d mysql redis minio minio-init`。
- 如需手动执行 Prisma db push，可显式注入根目录 `.env` 中的 `DATABASE_URL`，避免 workspace 下 Prisma CLI 读不到环境变量：

```powershell
$env:DATABASE_URL='mysql://xiaoman:password@localhost:3307/xiaoman_archive'; npm run prisma:dbpush
```
