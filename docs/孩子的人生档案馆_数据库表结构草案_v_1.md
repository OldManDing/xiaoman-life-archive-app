# 1. 文档说明

## 1.1 目标
本文档用于输出 MVP 阶段的数据库表结构草案，支撑以下工作：

- 后端建表与 ORM 模型设计
- 接口开发与联调
- 数据权限与查询逻辑设计
- 测试数据准备
- 后续扩展能力预留

## 1.2 设计原则
1. **先满足 MVP 闭环**，避免首期过度抽象。
2. **以逻辑删除为主**，保留后续恢复与审计空间。
3. **高频查询字段必须建索引**。
4. **媒体与主记录分表**，避免记录表过宽。
5. **家庭协作与孩子档案解耦**，支持一个用户管理多个孩子、多人协作同一孩子。
6. **AI 与月报结果独立存储**，避免重复计算影响主查询性能。

## 1.3 技术假设
本草案默认基于 MySQL 8.x 设计，字符集建议统一使用：

```sql
utf8mb4
```

时间字段建议统一使用：
- `created_at`
- `updated_at`
- `deleted_at`

金额字段如有支付接入，统一使用“分”为单位的整数存储。

# 2. 核心实体关系概览

建议 MVP 核心表如下：

1. `users` 用户表
2. `user_auth_accounts` 用户认证账号表
3. `children` 孩子档案表
4. `families` 家庭空间表
5. `family_members` 家庭成员表
6. `records` 记录主表
7. `record_media` 记录媒体表
8. `record_tags` 记录标签表
9. `milestone_templates` 里程碑模板表
10. `monthly_reports` 月报表
11. `monthly_report_records` 月报与记录关联表
12. `member_invites` 成员邀请表
13. `export_tasks` 导出任务表
14. `operation_logs` 关键操作日志表（建议）

# 5. 关键查询场景与索引理由

## 5.1 首页最近记录
查询条件：
- 按 `child_id`
- 按 `status=已发布`
- 按 `event_time desc`

因此 `records` 需要组合索引：
- `idx_child_id_status_event_time`

## 5.2 时间轴列表
查询条件：
- `child_id`
- `record_type`（可选）
- `status`
- `event_time` 排序

MVP 阶段可先依赖现有组合索引 + 条件过滤；如数据量上升，再补充：
- `idx_child_id_record_type_status_event_time`

## 5.3 按标签搜索
查询条件：
- `child_id`
- `tag_name`

因此 `record_tags` 需要：
- `idx_child_id_tag_name`

## 5.4 月报按月份查询
查询条件：
- `child_id`
- `month_key`

因此 `monthly_reports` 需要：
- `uk_child_month`

## 5.5 家庭成员查询
查询条件：
- `family_id`
- `status`

因此 `family_members` 需要：
- `uk_family_user`
- `idx_status`

# 7. 数据一致性建议

## 7.1 记录删除与月报重算
- 删除已发布记录后，不强制实时改写历史月报。
- 若月报重新生成，则按最新有效记录重算。

## 7.2 孩子删除与家庭关系
- 删除孩子档案时，不直接物理删除家庭空间。
- 若家庭下无有效孩子档案，可在后续治理任务中做归档。

## 7.3 成员移除
- 移除成员后，其历史创建的记录仍保留。
- 仅丧失后续访问权限。

## 7.4 媒体删除
- 记录删除后，媒体文件可延迟清理，避免误删影响恢复。
