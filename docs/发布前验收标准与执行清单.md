# 发布前验收标准与执行清单

更新时间：2026-05-28

## 通过判定

发布前验收按 P0 / P1 / P2 分级：

- P0：阻塞上线。必须修复并复验通过，不允许带问题发布。
- P1：影响体验或运营效率。默认应修复；如延期，必须写明原因、影响面和后续 owner。
- P2：上线后优化。不得影响核心链路、安全、数据正确性和付费/合规判断。

最终结论只能是以下三种：

- 通过：全部 P0 通过，P1 无明显风险或已记录延期方案。
- 条件通过：全部 P0 通过，但存在可控 P1 延期项。
- 不通过：任一 P0 未通过，或验收证据不足以证明关键链路可用。

## 验收证据要求

每一项验收必须至少保留一种证据：

- 自动化测试命令与结果。
- 真实浏览器截图或长截图。
- Android / iOS 真机或模拟器截图、录屏、网络响应记录。
- API 请求与响应摘要。
- 后台操作审计日志。

口头确认、只看代码、只看单页截图都不能作为 P0 通过依据。

## P1 验收项

| 编号 | 范围 | 标准 |
| --- | --- | --- |
| P1-01 | 搜索 | 搜索入口、历史搜索、热门标签、结果跳转可用，空状态有明确引导 |
| P1-02 | 月报/纪念册 | 页面不是纯占位；有故事化摘要、导出或申领反馈 |
| P1-03 | 地点 | 地点搜索、定位、手动输入均有反馈；真实地图 provider 上线前需通过登录后的 POI smoke，不能只看健康检查 provider 名称；后台系统运维页必须提示 `amap-regeo` 只算逆解析证据，不能替代 `source=amap` 的关键词 POI 候选 |
| P1-04 | 媒体格式 | JPG/PNG/WebP、MP4/WebM/MOV、M4A/MP3/WAV 等格式按设备能力验证 |
| P1-05 | iOS | iOS Safari 或 WebView 验证登录、注册、上传、定位、键盘和安全区；Web/Admin 源页面必须保留 `viewport-fit=cover`，让 `env(safe-area-inset-*)` 在 iOS 容器中生效 |
| P1-06 | 运维 | npm audit、生产 docker compose config、备份恢复演练、告警联系人确认；备份保留、恢复手册、演练时间和告警联系人需能在后台系统配置页查看或调整 |

## 发布前必跑命令

```bash
npm run verify:production-template
npm run verify:security-baseline
npm audit --omit=dev --audit-level=moderate
npm run regression:baseline
npm run build:api
npm run build:web
npm run build:admin
npm run test:e2e
docker compose -f deploy/docker-compose.prod.yml --env-file .env.example config --quiet
LIVE_TEST_USER_CREDENTIAL=<smoke-user> LIVE_TEST_USER_PASSWORD=<smoke-password> npm run verify:live-readiness
```

生产服务器或预发服务器上还必须执行 `npm run verify:production-env`。该命令会默认使用对象存储签名上传 URL 写入一段临时对象，检查浏览器上传所需 CORS 预检，再通过签名读取 URL 读回并清理临时对象；同时请求 AI `/chat/completions` 与高德 POI 接口验证 `STORAGE_*`、`AI_API_KEY`、`AI_MODEL`、`MAP_API_KEY` 是否真实可用。只有离线排查配置格式时才允许临时设置 `VERIFY_PRODUCTION_EXTERNAL_CHECKS=0`；正式上线验收不能跳过外部 provider 检查。`verify:live-readiness` 会登录测试账号后汇总 AI 预览和 POI 搜索两个子检查，任一 P0 失败都不允许宣称生产就绪；脚本只会对超时、网络错误和 502/503/504 等瞬时失败做有限重试，`INVALID_USER_KEY`、无效模型、provider 不一致或没有真实 POI 候选仍会记录为失败。若高德 Key 申请未完成，可显式设置 `LIVE_READINESS_ALLOW_P1_DEFERRALS=1`，仅在剩余失败全是 P1/P2 且已记录 owner 时生成“条件通过”报告；这不代表 P1-03 已完成。

## 验收执行顺序

1. 配置与工程门禁：生产模板、依赖审计、typecheck、单元测试、构建。
2. API 门禁：健康检查、注册邀请码、登录、注销账号后重新登录失败。
3. App 关键旅程：登录/注册/协议返回、首页、记录、时间轴、家庭、我的、账号注销。
4. 后台关键旅程：登录、总览、账号管理、邀请码、内容风险、客服反馈、列表详情、操作审计。
5. 视觉与产品审核：桌面、移动端、模拟器长截图；按 P0-17 / P0-18 判定。
6. 复验：所有修复项必须回到对应验收项重新跑证据。
